# 编码任务规则（所有Agent必须遵守）

## 目录管理规则
1. **新项目判断**：接到编码需求首先判断是否为新项目：
   - 新项目：主动询问用户项目名称，创建独立目录 `/home/admin/.openclaw/workspace-codingagent/projects/<项目名>/`
   - 迭代需求：直接在对应老项目目录下操作
   
2. **操作前告知**：每次开始编码/改bug前，必须明确告诉用户当前操作的目录路径，确认无误后再开始

3. **产物存放**：所有代码、文档、输出产物全部放在对应项目目录下，禁止乱放文件

## 新需求开发流程（Bug修复无需走此流程）
所有新需求必须严格按照「需求确认→设计→任务拆分→执行」的流程推进：
1. **第一步：需求确认**
   - 接到需求后先梳理成 `spec.md`（需求规格说明），包含功能描述、预期效果、边界条件
   - 发给用户确认，用户完善后进入下一步

2. **第二步：设计确认**
   - 基于确认后的spec生成 `design.md`（技术设计），包含技术选型、架构设计、模块划分、接口定义
   - 发给用户确认，设计通过后进入下一步

3. **第三步：任务拆分**
   - 基于设计生成 `tasks.md`（任务列表），拆分成多个可独立执行的子任务，每个任务包含：任务描述、预计耗时、状态（待处理/进行中/已完成）
   - 执行过程中动态更新tasks.md的任务状态，用户可随时查看进度

4. **第四步：执行开发**
   - 按tasks.md的任务顺序开发，完成一个标记一个
   - 全部完成后统一交付给用户验收

## 编码规范
- 代码必须加清晰注释
- 每个项目必须包含 README.md 说明使用方法
- 敏感信息（密钥、密码等）必须单独放在 config.js/.env 配置文件，禁止硬编码
- 提交前必须测试可正常运行

## 测试规范
- 所有任务必须包含测试环节，测试覆盖三类：
  1. 检查点验证：每个任务完成后必须通过增量检查点验证，确保当前功能正确
  2. 属性测试：验证通用正确性属性，适配不同场景和边界
  3. 单元测试：验证具体示例和边界情况，覆盖异常场景
- 项目上线前必须通过全量功能测试和兼容性测试

## API限流机制（大模型调用规则）
- 大模型API请求限制：5小时内最多请求N次，避免触发限流
- 心跳检测机制：
  1. 每次心跳时简单测试API联通性，最多重试3次
  2. 联通失败则暂停请求，等待下次心跳再试，不重复请求
  3. 联通成功后，自动检查子Agent未完成的任务，恢复执行
  4. 所有任务已完成则不做任何操作，避免不必要请求
